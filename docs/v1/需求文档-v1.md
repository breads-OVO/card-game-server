## 1. 项目概述

### 1.1 项目背景

构建一个分布式、高性能的卡牌游戏服务器，支持海量玩家同时在线进行卡牌对战。系统采用微服务架构设计，各服务职责单一、独立部署，通过 gRPC 进行服务间通信，使用 etcd 实现服务注册与发现。

### 1.2 项目目标

1. 提供稳定的客户端 TCP 长连接接入能力
2. 实现用户认证鉴权体系（注册、登录、Token 管理）
3. 实现玩家在线会话管理
4. 构建可扩展的消息路由框架
5. 支持未来业务模块的快速接入

### 1.3 适用范围

本文档覆盖游戏服务器后端所有模块的需求定义，适用于开发、测试、运维团队。

### 3.1 Gateway 网关服务

#### 3.1.1 功能列表

| 编号 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| GATEWAY-001 | TCP 长连接管理 | P0 | 基于 Netty 管理客户端 TCP 长连接，支持大量并发连接 |
| GATEWAY-002 | Protobuf 编解码 | P0 | 支持 Varint32 长度帧 + Protobuf 消息的编解码 |
| GATEWAY-003 | 消息路由分发 | P0 | 根据 MessageType 前缀（AUTH_/AGENT_/HEART_）自动分发到对应处理器 |
| GATEWAY-004 | 心跳检测 | P0 | 读空闲超时检测（默认 90s），自动关闭僵尸连接 |
| GATEWAY-005 | 连接会话管理 | P0 | 管理 Channel 与 PlayerId 的映射关系 |
| GATEWAY-006 | 断连通知 | P0 | 连接断开时自动通知 Agent 服务清理会话 |
| GATEWAY-007 | 认证消息处理 | P0 | 处理登录、注册、登出请求，转发到 Author 服务 |
| GATEWAY-008 | Agent 消息处理 | P0 | 处理玩家上下线通知、消息转发、状态查询、广播、踢下线等请求 |
| GATEWAY-009 | 心跳回复 | P0 | 收到客户端心跳后回复服务端时间戳 |
| GATEWAY-010 | 健康检查 | P1 | 提供 HTTP 健康检查接口，返回连接数、在线玩家数等状态 |

#### 3.1.2 详细需求

**GATEWAY-003 消息路由**

消息路由基于 `MessageHandlerFactory` 实现，采用前缀匹配策略：

| 消息类型前缀 | 处理器 | 目标服务 |
|-------------|--------|---------|
| `AUTH_` | AuthMessageHandler | Author 服务（gRPC） |
| `AGENT_` | AgentMessageHandler | Agent 服务（gRPC） |
| `HEART_` | HeartBeatMessageHandler | 本地处理 |

**GATEWAY-005 会话管理**

`ChannelSessionManager` 维护以下映射关系：

| 映射 | 类型 | 说明 |
|------|------|------|
| ChannelId → Channel | 连接注册表 | 所有已连接的 Channel |
| PlayerId → Channel | 玩家绑定表 | 登录后绑定 PlayerId 与 Channel |
| ChannelId → PlayerId | 反向绑定表 | 根据 Channel 反查 PlayerId |

**GATEWAY-006 断连流程**

```
连接断开 → channelInactive()
  ├── 若已绑定 PlayerId：
  │    1. AgentGrpcClient.notifyPlayerOffline() → Agent 清理会话
  │    2. sessionManager.removeChannel()
  └── 若未绑定 PlayerId：
        sessionManager.removeChannel()
```

### 3.5 协议定义

#### 3.5.1 通信协议

所有通信采用两层协议：

1. **传输层**：Netty 帧协议（Varint32 长度前缀 + Protobuf 二进制数据）
2. **消息层**：统一 `GameMessage` 包装

```protobuf
message GameMessage {
  MsgHeader header = 1;           // 消息头
  bytes body = 2;                 // 消息体（具体业务数据）
  MessageType messageType = 3;    // 消息类型标识
}

message MsgHeader {
  string msgId = 1;       // 消息ID (UUID)
  int32 version = 2;      // 协议版本（当前为 1）
  int64 timestamp = 3;    // 时间戳
  string seqId = 4;       // 请求序列号（用于请求-响应匹配）
  string token = 5;       // 认证 Token
  int32 compress = 6;     // 压缩标志（0=不压缩, 1=gzip）
}
```

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 并发连接数 | ≥ 100,000 | Gateway 单节点支持的 TCP 长连接数 |
| 在线玩家数 | ≥ 100,000 | Agent 单节点管理的在线玩家数 |
| 消息吞吐量 | ≥ 10,000 TPS | Gateway 消息处理能力 |
| 认证响应时间 | ≤ 200ms (P99) | Author 服务认证接口响应时间 |
| 消息转发延迟 | ≤ 50ms (P99) | Agent 消息转发延迟 |

### 4.2 可用性需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 系统可用性 | ≥ 99.9% | 全年停机时间不超过 8.76 小时 |
| 服务注册 | 服务启动后 5s 内完成注册 | etcd 注册延迟 |
| 故障恢复 | ≤ 30s | 服务宕机后被及时发现并从路由表中移除 |

### 4.3 可扩展性需求

1. **水平扩展**：Gateway、Agent、Author 均支持多实例部署，通过 etcd 实现动态服务发现
2. **业务模块扩展**：新增业务服务只需实现对应 gRPC 接口，并在 Gateway 中添加消息处理器
3. **协议扩展**：新增消息类型直接在 `MessageType` 枚举中添加，不影响现有协议

### 4.4 安全性需求

| 需求 | 说明 |
|------|------|
| 密码加密 | 密码使用 BCrypt 等安全哈希算法存储 |
| Token 鉴权 | 使用 JWT 无状态 Token，支持过期和刷新 |
| Token 缓存 | Redis 缓存 Token，支持登出后立即失效 |
| 连接鉴权 | 未登录的连接只能发送认证类消息 |
| 参数校验 | 所有外部输入进行参数合法性校验 |

## 8. 消息流场景

### 8.1 客户端连接与认证

```
Step 1: Client 建立 TCP 连接 → Gateway
Step 2: Client 发送 AUTH_REGISTER_REQUEST / AUTH_LOGIN_REQUEST
Step 3: Gateway 路由到 AuthMessageHandler
Step 4: AuthMessageHandler 调用 AuthorGrpcClient 对应方法
Step 5: Author 处理业务逻辑（注册账号/验证密码签发 Token）
Step 6: Response 返回 Gateway → 转发给 Client
Step 7: (登录成功后) Gateway 调用 AgentGrpcClient.notifyPlayerOnline()
Step 8: Agent 记录玩家会话
```

### 8.2 心跳维护

```
定时流程：
  Client 每隔 30s 发送 HEART_BEAT
    → HeartBeatMessageHandler 回复 CommonResponse(SUCCESS, 时间戳)

异常流程：
  Gateway 90s 内未收到任何数据
    → HeartbeatHandler 触发 READER_IDLE 事件
    → ctx.close() 关闭连接
    → ServerChannelHandler.channelInactive()
    → AgentGrpcClient.notifyPlayerOffline()
    → sessionManager.removeChannel()
```

### 8.3 断连重连

```
Client 异常断连：
  → Gateway channelInactive()
  → Agent 清理会话
  → PlayerId 与 Channel 解绑

Client 重连：
  → 新 TCP 连接
  → 发送 AUTH_LOGIN_REQUEST（带已有 Token）
  → Author 验证 Token + Agent 刷新会话
```

## 10. 附录

### 10.1 术语表

| 术语 | 说明 |
|------|------|
| Gateway | 网关服务，客户端 TCP 接入层 |
| Agent | 会话管理服务，消息路由中间件 |
| Author | 认证服务，账号管理 |
| PlayerId | 玩家唯一标识 |
| Token | JWT 无状态认证令牌 |
| Channel | Netty 网络连接通道 |
| Session | 玩家在线会话 |
| etcd | 分布式键值存储，用于服务注册与发现 |
| Proto | Protobuf 协议文件 |

### 10.2 工程依赖关系

```
card-game-server (pom)
├── cd-proto        — Protobuf 定义 + gRPC 服务定义
├── cd-common       — 公共工具、配置、gRPC 客户端、etcd 服务发现/注册
├── cd-gateway      — Netty 网关（依赖 cd-common, cd-proto）
├── cd-author       — 认证服务（依赖 cd-common, cd-proto）
└── cd-agent        — Agent 服务（依赖 cd-common, cd-proto）
```
