## 1. 项目概述

### 1.1 项目背景

v1.0 已完成基础网络接入（Gateway）、会话管理（Agent）、认证鉴权（Author）等基础设施。v2.0 的目标是实现类似三国杀的卡牌对战核心玩法，包括玩家匹配、房间管理、回合制战斗、技能交互、对局结算等，构建可水平扩展的匹配服务与游戏逻辑服务。

### 1.2 项目目标

1. **匹配服务**：提供基于段位分的快速匹配能力，支持匹配队列管理、超时处理、匹配成功回调
2. **游戏逻辑服务**：管理游戏房间，执行完整的类三国杀回合制规则，处理出牌、技能、伤害、死亡等逻辑，实时同步游戏状态
3. **与 v1 服务无缝集成**：Gateway 新增 MATCH\_ 和 GAME\_ 消息处理分支，复用现有路由框架和会话管理能力
4. **支持水平扩展**：匹配和游戏服务均支持多实例部署，通过 etcd 实现服务发现
5. **容灾与断线重连**：确保游戏过程的高可用与玩家体验

### 1.3 适用范围

本文档覆盖匹配服务、游戏逻辑服务的详细需求，包括匹配流程、房间生命周期、游戏状态机、协议定义及容灾策略。与 v1 文档共同构成完整的后端系统说明。

### 2.2 核心通信路径

**v2.0 所有客户端消息统一通过 Gateway 路由，不再经过 Agent 转发业务消息：**

```
客户端 → Gateway(路由层) ─┬── AUTH_xxx  → AuthMessageHandler → Author(gRPC)
                          ├── AGENT_xxx → AgentMessageHandler → Agent(gRPC)
                          ├── MATCH_xxx → MatchMessageHandler → Matching(gRPC)
                          ├── GAME_xxx  → GameMessageHandler  → Game(gRPC)
                          └── HEART_xxx → HeartBeatMessageHandler(本地)
```

> **设计决策**：Agent 的 `ForwardMessage` 接口仅在 "其他业务服务需要主动推送消息给指定玩家" 的场景使用（如 Game 回调通知），客户端到业务服务的请求不经过 Agent 中转，降低链路延迟。

## 4. 匹配服务详细设计

### 4.1 匹配状态机

```
                          ┌──────────┐
                          │   空闲    │
                          └────┬─────┘
               ┌──────────────┼──────────────┐
               │ 发起匹配      │              │ 取消匹配
               ▼              │              │
          ┌──────────┐        │              │
          │  匹配中   │────────┼──────────────┘
          └────┬─────┘        │
       ┌───────┼───────┐      │
       │ 超时   │ 成功   │     │
       ▼       ▼        │     │
  ┌────────┐ ┌──────────┐     │
  │匹配失败│ │等待创建房间│─────┘
  └────────┘ └────┬─────┘
                  │ Game.CreateRoom 结果
            ┌─────┴──────┐
            │ 成功        │ 失败(重试3次)
            ▼             ▼
        ┌──────────┐ ┌──────────┐
        │匹配完成   │ │匹配失败  │
        └──────────┘ └──────────┘
```

### 4.2 匹配算法

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 目标人数 | 8 | 经典 8 人局，v2.0 固定为 8 人 |
| 初始段位分范围 | ±200 | 使用简化 ELO 分 |
| 扩圈策略 | 每 10 秒扩大 100 分 | 上限 ±1000，超过后不再扩圈 |
| 最大等待时间 | 120 秒 | 超时自动失败 |
| 队列扫描间隔 | 2 秒 | 后台定时任务触发 |
| 初始段位分 | 1200 | 新玩家默认分 |

**匹配匹配算法逻辑**：

```
输入：gameMode, 当前时间

1. 获取队列中所有玩家分数分布
2. 若队列人数 < 8, 返回空
3. 从队列头取最高分段作为基准
4. 计算当前允许的分数范围 = 初始范围 + min(扩圈次数 × 每步扩圈, 最大范围)
5. 从队列中找出在 [基准 - 范围，基准 + 范围] 内分数最高的 8 人
6. 若找到 8 人，从队列移除并调用 Game.CreateRoom
7. 若 Game 创建失败（重试 3 次），将 8 人重新入队
```

### 4.3 匹配队列管理

- **存储**：Redis Sorted Set，key = `match:queue:{gameMode}`，member = `playerId`，score = 段位分
- **取消匹配**：从队列中删除玩家，若已在匹配中但未完成组队，直接移除并回复取消成功
- **重复匹配限制**：同一玩家不能同时发起多次匹配，需先取消再发起
- **分布式锁**：使用 Redisson RLock 防止多实例并发扫描导致同一玩家被匹配多次

### 4.4 段位分管理

**段位分更新归属：Matching 服务**

```
Game 结算完成 → Game.UpdateRating(gRPC) → Matching 更新段位分
```

- **幂等性**：使用 `gameRoundId` 作为幂等键，Redis 记录已处理的 gameRoundId，有效期 24 小时
- **更新公式**（简化 ELO）：`newScore = oldScore + K × (actual - expected)`，其中 `K = 32`，`expected = 1 / (1 + 10^((opponentAvgScore - myScore) / 400))`
- 获胜方 `actual = 1`，失败方 `actual = 0`
- 段位分变更记录写入 MySQL（`player_rating_log` 表，用于历史查询）

### 4.5 gRPC 服务定义 — MatchingService


## 5. 游戏逻辑服务详细设计

### 5.1 房间生命周期

```
匹配成功(Game.CreateRoom)
         │
         ▼
    ┌──────────┐
    │  WAITING  │  等待玩家确认加载（30s 超时）
    └─────┬────┘
          │ 全部确认 / 倒计时结束
          ▼
    ┌──────────┐
    │ PLAYING   │  正常对局
    └─────┬────┘
          │ 仅剩一方存活
          ▼
    ┌──────────┐
    │ SETTLEMENT│  结算（写入 MySQL，更新段位分）
    └─────┬────┘
          │ 结算完成
          ▼
    ┌──────────┐
    │ DESTROYED│  释放房间资源
    └──────────┘
```

| 状态 | 说明 | 超时/异常处理 |
|------|------|-------------|
| WAITING | 房间创建后等待玩家进入，玩家需发送 `JOIN_ROOM_REQUEST` 确认 | 30 秒未确认视为逃跑（逃跑者扣分），房间进入 SETTLEMENT |
| PLAYING | 正常对局，执行回合制逻辑 | 玩家断线进入托管；若所有玩家均断线，房间冻结 30 分钟后销毁 |
| SETTLEMENT | 计算胜负、MVP、积分变化，写入 MySQL，调用 Matching.UpdateRating | 必须完成结算，即使部分玩家断线。MySQL 写入失败重试 3 次 |
| DESTROYED | 从内存和 Redis 中删除房间数据 | 无 |

### 5.2 游戏状态机（回合制）

#### 5.2.1 回合阶段（Turn Phase）

```
[回合开始]
    │
    ▼
[准备阶段]  技能触发点（如诸葛观星）
    │
    ▼
[判定阶段]  判定区延时锦囊（乐不思蜀、兵粮寸断）
    │
    ▼
[摸牌阶段]  默认摸 2 张，技能可改变数量
    │
    ▼
[出牌阶段]  核心阶段：使用杀、锦囊、装备、技能，可主动结束
    │
    ▼
[弃牌阶段]  手牌数 > 体力值则弃至体力值
    │
    ▼
[结束阶段]  技能触发点（如闭月）
    │
    ▼
[回合结束]  移交下一个玩家
```

#### 5.2.2 响应链模型

类三国杀的技能响应采用 **Request-Response** 模型，系统进入响应等待时暂停当前阶段：

```
玩家 A 出"杀"指定玩家 B
  → 系统状态变更为 RESPONDING(等待 B 出"闪")
  → 向 B 推送 ACT_REQUEST(action="闪", timeout=5s)
  → B 选择出"闪" → 杀无效，回到出牌阶段
  → B 不出或超时 → 杀命中，B 扣血
  → 若扣血后 B 体力 ≤ 0，进入濒死流程：
    系统状态变更为 RESPONDING(等待求桃)
    向 B 自身及其队友推送 ACT_REQUEST(action="桃")
    若 5 秒内无人出桃，B 死亡
    若有桃，B 恢复至 1 血
  → 回到出牌阶段
```

**群攻锦囊（南蛮入侵、万箭齐发）**：按座位顺序依次要求每个玩家响应，链式执行。

#### 5.2.3 核心玩法要素

| 要素 | v2.0 实现范围 |

### 5.4 游戏引擎实现方案

每个房间使用 **单线程 Actor 模型**，避免并发问题：

### 5.5 状态同步策略

| 同步类型 | 时机 | 内容 | 推送方式 |
|---------|------|------|---------|
| 全量同步 | 进入房间、断线重连 | `GAME_SNAPSHOT` — 完整的游戏状态（手牌内容、武将、装备、牌堆等） | 单播给目标玩家 |
| 增量同步 | 每次操作后 | `GAME_STATE_UPDATE` — 变化的字段（手牌数、体力、装备变化、卡牌事件、伤害事件、阶段变更） | 广播给房间内所有玩家 |
| 行动请求 | 需要玩家操作时 | `ACT_REQUEST` — 要求指定玩家出牌/出技能/响应 | 单播给待操作玩家 |

**快照持久化**：每个关键状态变更后异步写入 Redis，key = `game:room:{roomId}:snapshot`（JSON），过期时间 30 分钟。不阻塞主流程。

## 7. 房间管理

### 7.1 房间创建与分配

- **触发时机**：Matching 服务匹配成功后，调用 Game 服务的 `CreateRoom` RPC
- **负载均衡**：Matching 从 etcd 获取所有健康 Game 实例列表，轮询或选取房间数最少的实例
- **重试**：若 `CreateRoom` 失败，更换实例重试最多 3 次

### 7.2 房间内玩家管理

| 操作 | 处理方式 |
|------|---------|
| 玩家正常退出 | 游戏未开始时允许退出，游戏中不允许（只能逃跑） |
| 玩家逃跑 | 标记为死亡，弃置所有牌，触发死亡惩罚；逃跑者扣段位分，其余玩家继续游戏 |
| 玩家断线 | 标记为 OFFLINE 状态，进入托管模式（自动使用基本牌响应，不出锦囊和装备）；断线超过 90 秒未重连视为逃跑 |
| 玩家重连 | 清除 OFFLINE 标记，恢复控制权，下发完整快照 |

### 7.3 房间与 Game 实例映射

- 房间创建后，Game 实例在本地内存中维护房间 Actor，同时在 Redis 记录 `game:room:{roomId}:owner → instanceId`
- 若 Game 实例宕机，其他实例可读取 Redis 中的房间快照尝试恢复

### 7.4 房间状态持久化

| 存储 | Key 格式 | 内容 | 过期时间 |
|------|---------|------|---------|
| Redis | `game:room:{roomId}:info` | Hash — 房间元数据（状态、玩家列表、创建时间） | 30 min |
| Redis | `game:room:{roomId}:snapshot` | String — 完整游戏状态 JSON | 30 min |
| Redis | `game:room:{roomId}:events` | List — 最近 100 条事件日志（用于重连回放） | 30 min |
| Redis | `game:room:{roomId}:owner` | String — 所在 Game 实例 ID | 30 min（带租约） |
| MySQL | `game_record` | 对局记录（房间 ID、开始/结束时间、玩家、胜负、积分变化） | 永久 |
| MySQL | `player_rating_log` | 段位分变更流水（playerId、变化前后分数、gameRoundId） | 永久 |
| MySQL | `game_event_log`（可选） | 操作流水（用于审计和回放） | 按需保留 |

## 8. 断线重连流程

```
1. Client 检测到 TCP 断开
2. Client 重新建立 TCP 连接至 Gateway
3. Client 发送 AUTH_LOGIN_REQUEST（带旧 Token）重新认证
4. Agent 刷新会话，绑定新 Channel
5. Client 发送 RECONNECT_REQUEST (roomId, playerId)
6. Gateway → GameMessageHandler → GameGrpcClient.getSnapshot()
7. Game 校验：
   - 玩家确实在该房间内
   - 玩家状态为 OFFLINE
8. Game 从 Redis 获取最新快照，构建 GAME_SNAPSHOT 消息
9. 通过 Gateway 向该玩家单播 GAME_SNAPSHOT（含完整状态 + 最近事件历史）
10. Client 本地重建状态
11. Game 将玩家状态置为 ALIVE，取消托管，通知房间其他玩家
```

## 9. 容灾策略

### 9.1 匹配服务容灾

| 故障场景 | 处理策略 |
|---------|---------|
| 实例宕机 | 无状态设计，队列在 Redis 中。其他实例可继续处理 |
| Redis 不可用 | 匹配服务停止匹配并告警，已有请求返回匹配失败 |
| 客户端超时重试 | 客户端匹配请求超时（5s）可重试，服务端用 playerId + gameMode 去重 |

### 9.2 游戏服务容灾

| 故障场景 | 处理策略 |
|---------|---------|
| Game 实例宕机 | 1. etcd 剔除该实例，Matching 不再向其分配房间<br>2. 其他实例从 Redis 读取快照尝试恢复房间（5-10s）<br>3. 恢复失败则广播"对局异常终止"，释放房间<br>4. 记录异常日志，人工介入补偿玩家积分 |
| 玩家断线 | 见第 8 章重连流程。超时 90s 未重连视为逃跑 |
| Redis 不可用 | 降级为仅内存模式，但实例宕机时无法恢复。暂停新游戏创建 |
| Actor 死锁/卡死 | 看门狗线程：若一个操作超过 30s 未返回，强制 kill 房间 Actor，清理房间并通知玩家 |

### 9.3 数据持久化

- **对局记录**：Game 结算时必须成功写入 MySQL，失败重试 3 次，仍失败则写入本地文件并告警
- **段位分更新**：调用 Matching 的 `UpdateRating` 允许失败重试，使用 `gameRoundId` 幂等
